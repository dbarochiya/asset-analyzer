### 🔢 Per-Row Calculations

| Field | Formula | Correctness | Usefulness | Score | Critical Issue |
|-------|---------|-------------|------------|-------|---------------|
| **total_invested** | Running cumsum of `cashflow` | ✅ | ✅ | **90/100** | Correct money invested over time. Minor: naming collision — "total invested" sounds like current at-risk capital, but it's actually cumulative net cashflows including withdrawals. |
| **total_pv** | `pv + free_cash` | ✅ | ✅ | **92/100** | Clean. |
| **gain_unrealised** | `pv − invested` | ✅ | ✅ | **82/100** | "Invested" here is the *current invested value* (column in spreadsheet), not cumulative. This is correct but relies entirely on user inputting correct "Invested" values. If user inputs are stale, this is wrong. No validation. |
| **gain_realised** | `invested + free_cash − cumInvested` | ✅ | ✅ | **78/100** | This is the *book profit already taken out*. Conceptually correct — it's what you've pocketed. BUT: it can go negative early in an account's life when you haven't withdrawn anything yet (cumInvested grows while invested+freecash stays small). This negative "realised" gain is confusing — it just means "you've put in more than you've taken out so far". |
| **gain_total** | `gain_unrealised + gain_realised` = `total_pv − total_invested` | ✅ | ✅ | **92/100** | Pure and correct. |
| **returns_pct** | `gain_total / cumInvested × 100` | ✅ | ⚠️ Partially | **68/100** | ❌ **Critical issue**: This is a **simple return** (not annualised). For portfolios running 5+ years, a 20% return looks the same whether earned in 1 year or 10 years. Users often mis-read this as a meaningful performance indicator when it's really just a raw gain percentage. Should be labelled clearly as "Cumulative Return %" not just "Returns %" |
| **NAV (per row)** | Mutual-fund unit method: first row sets `units = total_pv/100`, subsequent rows: `units += cashflow/navPrev`, `nav = total_pv/units` | ✅ | ✅ | **85/100** | Time-weighted return proxy is conceptually correct. Issue: if `navPrev = 0` (completely empty portfolio), new units would be infinite. The code handles this with `navPrev > 0 ? cf/navPrev : 0` which silently drops the cashflow — acceptable but unauditable. |
| **monthly_xirr_pct** | XIRR recalculated at every row using all prior rows | ✅ | ✅ | **72/100** | ⚠️ **Misleading label**: This is not "Monthly XIRR" — it is the *annualised XIRR up to this point in time*. Monthly XIRR would be ~XIRR/12. The column header causes confusion. Should be renamed to "Running XIRR (Annualised)". Also: this is O(n²) — expensive for large portfolios. |

### 📊 Analytics Computed Fields

| Field | Formula | Correctness | Usefulness | Score | Notes |
|-------|---------|-------------|------------|-------|-------|
| **Max Drawdown** | [(peak_pv − current_pv) / peak_pv × 100](file:///Users/dhruvbarochiya/Workspace/analyzer/parse_data.py#24-26) on total_pv series | ✅ | ✅ | **80/100** | Drawdown on *total PV* is correct. Issue: drawdown is capped at exactly 100% — when account zeroes out, this always reports 100%, obscuring any meaningful intra-period drawdown. |
| **Best / Worst Month** | Argmax/argmin of `returns_pct` series | ⚠️ Partially | ✅ | **60/100** | ❌ **Correctness issue**: Best/Worst month is from the `returns_pct` series which is a **cumulative return** not a **monthly return**. It finds the month where total cumulative return was highest, not the best single month's performance. The correct field to use is `monthly_returns` (month-over-month % change in total_pv). This makes "best month: 2020-12 +28.57%" actually the best *cumulative total return* that month, not the best monthly performance. |
| **Max Consecutive Neg. Months** | Count of consecutive `returns_pct < 0` | ❌ Wrong variable | ⚠️ | **40/100** | ❌ Same issue as above — uses cumulative returns not monthly returns. A month's cumulative return can go negative even if the portfolio gained value that month if early losses dominate. Should use month-over-month PV change. |
| **Breakeven Date** | First date returns_pct flips from negative to non-negative | ⚠️ | ✅ | **65/100** | Same issue: uses cumulative returns. However the use case (when did you first get back to break-even overall) is arguably the right one — you want the total cumulative return to flip positive. Label is fine, the underlying metric matches the question. |
| **Avg Monthly Return %** | Mean of `returns_pct` series | ❌ Incorrect | ⚠️ | **35/100** | ❌ **Mathematically wrong**: Averaging cumulative returns across periods is meaningless. Month 3's cumulative 5% and Month 6's cumulative 8% — averaging gives 6.5% but that tells you nothing about actual monthly performance. Should average [(r_t − r_{t-1})](file:///Users/dhruvbarochiya/Workspace/analyzer/parse_data.py#24-26) = month-over-month changes. |
| **Volatility %** | Standard deviation of `returns_pct` series | ❌ Incorrect | ⚠️ | **35/100** | ❌ Same issue — standard deviation of *cumulative* returns is a non-standard, misleading volatility measure. It will always increase monotonically over time as cumulative returns diverge. Correct volatility is std dev of *monthly* returns (period-over-period), then optionally annualised via `× √12`. |

## Part 4 — Critical Bugs to Fix

> [!CAUTION]
> The following are actual mathematical errors that produce wrong outputs, not just display issues.

1. **Best/Worst Month** uses cumulative returns instead of monthly returns → wrong answer
2. **Avg Monthly Return %** averages cumulative series → meaningless number
3. **Volatility %** = std dev of cumulative series → always grows, not real volatility
4. **Max Consecutive Negative Months** uses cumulative not periodic → wrong count

**Fix**: Add a derived field `monthly_return_pct` computed as [(total_pv_t / total_pv_{t-1} − 1) × 100](file:///Users/dhruvbarochiya/Workspace/analyzer/parse_data.py#24-26) and use it for all the above.

## Part 5 — Missing Features & Ideas 💡

### High Value (implement first)

| Feature | Why Useful |
|---------|-----------|
| **Monthly Return % (periodic)** | The most fundamental portfolio metric — what did I actually gain/lose THIS month? Currently missing; `returns_pct` is cumulative. |
| **Sharpe Ratio** | Risk-adjusted return = [(XIRR − risk_free_rate) / annualised_volatility](file:///Users/dhruvbarochiya/Workspace/analyzer/parse_data.py#24-26). Makes benchmarking meaningful — a 15% XIRR with high volatility may be worse than a 12% XIRR that's steady. |
| **Account Correlation Matrix** | If you have zerodha + paytm-mf, are they moving together? If correlation is high, you have false diversification. A heatmap shows this instantly. |
| **Cashflow-Weighted vs Time-Weighted Toggle** | Let user switch between XIRR (money-weighted) and NAV (time-weighted) as the *primary* KPI, side by side, so the impact of their investment timing is immediately visible. |
| **Rolling 12-Month XIRR** | Instead of cumulative XIRR (gets more stable over time), show XIRR of the *last 12 months* as a moving window. This tells you "how am I doing recently" not "how did I do historically". |

### Medium Value

| Feature | Why Useful |
|---------|-----------|
| **IRR per Investment Tranche** | If you do SIPs, each monthly investment has its own holding period. Show the IRR of each individual tranche to see which buys were good/bad. |
| **Tax Lot Tracking** | For equity, LTCG kicks in after 1 year. Highlight which lots are short-term vs long-term. |
| **Goal Tracker** | User sets target corpus (e.g. ₹50L by 2030) and sees if they're on track given current XIRR trajectory. |
| **Inflation-Adjusted Returns** | Show XIRR net of CPI (India ~6%). A 10% XIRR with 6% inflation = only 4% real return. |
| **Reinvestment Rate Assumption** | When modelling future value, free cash is assumed to earn 0%. Allow user to set a reinvestment rate for idle cash. |

### Nice to Have

| Feature | Why Useful |
|---------|-----------|
| **Benchmark XIRR** | Not just NAV comparison but "if you had invested the same cashflows in Nifty, what XIRR would you have gotten?" Uses same dates & amounts as your actual cashflows. |
| **Portfolio Concentration** | How much of your total PV is in each account? A pie/treemap chart. |
| **Annual Summary View** | Aggregate by year — XIRR per year, gain per year. Easier to digest than monthly. |
| **Export to PDF** | Single-page report card with all KPIs and charts. |
| **Scenario Analysis** | "What if my portfolio dropped 20% from here?" — show impact on XIRR and gain. |
